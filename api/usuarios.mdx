The Users API controls who can log in to the system and what they can do. Access control is organized into three layers: **users** (credentials + linked employee), **roles** (named permission sets), and **modules** (navigation items available to each role). Permissions are stored as a bitmask in the `permisos` field of each role.

<Note>
  All endpoints require a valid Bearer token unless otherwise noted. Include `Authorization: Bearer <token>` in every request header.
</Note>

---

## Users

### List active users

`GET /api/usuarios`

Returns all users where `vigente = true`.

<CodeGroup>

```bash cURL
curl --request GET \
  --url 'https://your-api.example.com/api/usuarios' \
  --header 'Authorization: Bearer <token>'
```

</CodeGroup>

#### Response fields

<ResponseField name="id" type="number">Unique user ID.</ResponseField>
<ResponseField name="nombre" type="string">Display name.</ResponseField>
<ResponseField name="dni" type="string">Identity document number.</ResponseField>
<ResponseField name="email" type="string">Login email address.</ResponseField>
<ResponseField name="vigente" type="boolean">`true` if the account is active.</ResponseField>
<ResponseField name="rol" type="object">
  Assigned role.
  <Expandable title="RolResponse properties">
    <ResponseField name="id" type="string">Role identifier (e.g., `ADMIN`, `TECNICO`).</ResponseField>
    <ResponseField name="nombre" type="string">Human-readable role name.</ResponseField>
    <ResponseField name="descripcion" type="string">Role description.</ResponseField>
    <ResponseField name="permisos" type="number">Bitmask integer encoding the set of permissions granted. See [Permission bitmask reference](#permission-bitmask-reference).</ResponseField>
  </Expandable>
</ResponseField>
<ResponseField name="empleado" type="object">
  Linked employee record. See [EmpleadoResponse](/api/empleados) for field details.
</ResponseField>

### Create a user

`POST /api/usuarios`

Creates a new user account linked to an existing employee.

<ParamField body="nombre" type="string" required>
  Display name for the user.
</ParamField>

<ParamField body="dni" type="string" required>
  Identity document number.
</ParamField>

<ParamField body="email" type="string" required>
  Login email address. Must be unique.
</ParamField>

<ParamField body="password" type="string" required>
  Initial password. Stored as a bcrypt hash.
</ParamField>

<ParamField body="idRol" type="string" required>
  Role ID to assign (e.g., `ADMIN`, `TECNICO`). Use `GET /api/usuarios/roles` to retrieve valid IDs.
</ParamField>

<ParamField body="empleadoId" type="number" required>
  ID of the employee record to link. Each employee can only be linked to one active user.
</ParamField>

<CodeGroup>

```bash cURL
curl --request POST \
  --url 'https://your-api.example.com/api/usuarios' \
  --header 'Authorization: Bearer <token>' \
  --header 'Content-Type: application/json' \
  --data '{
    "nombre": "Luis Torres",
    "dni": "72345678",
    "email": "ltorres@empresa.pe",
    "password": "SecurePass123!",
    "idRol": "TECNICO",
    "empleadoId": 5
  }'
```

</CodeGroup>

## Roles

Roles define what actions a user is permitted to take. The `permisos` field is a bitmask — an integer computed by adding together individual permission values.

---

### List active roles

`GET /api/usuarios/roles`

Returns all roles that are currently active.

<CodeGroup>

```bash cURL
curl --request GET \
  --url 'https://your-api.example.com/api/usuarios/roles' \
  --header 'Authorization: Bearer <token>'
```

</CodeGroup>

---

### Create a role

`POST /api/usuarios/roles`

Creates a new role with a set of permissions encoded as a bitmask.

<ParamField body="id" type="string" required>
  Unique string identifier for the role (e.g., `TECNICO`, `SUPERVISOR`). Used as the foreign key in user records.
</ParamField>

<ParamField body="nombre" type="string" required>
  Human-readable role name.
</ParamField>

<ParamField body="descripcion" type="string">
  Description of the role's purpose.
</ParamField>

<ParamField body="permisos" type="number" required>
  Bitmask integer encoding the granted permissions. Sum the values of all desired permissions. See [Permission bitmask reference](#permission-bitmask-reference).
</ParamField>

<CodeGroup>

```bash cURL
curl --request POST \
  --url 'https://your-api.example.com/api/usuarios/roles' \
  --header 'Authorization: Bearer <token>' \
  --header 'Content-Type: application/json' \
  --data '{
    "id": "TECNICO",
    "nombre": "Técnico",
    "descripcion": "Field technician with execution rights",
    "permisos": 37
  }'
```

</CodeGroup>

<Tip>
  `permisos: 133` = `REGISTRAR_PENDIENTE` (1) + `VER_DETALLE_PENDIENTE` (4) + `COMENZAR_TRABAJO` (128) = 133. See the full bitmask table below.
</Tip>

---

</ParamField>

<CodeGroup>

```bash cURL

  --url 'https://your-api.example.com/api/usuarios/roles/TECNICO' \
  --header 'Authorization: Bearer <token>' \

    "permisos": 101

```

</CodeGroup>

---

### Delete a role

`DELETE /api/usuarios/roles/{id}`

Deletes a role. **This operation is blocked if any active users are currently assigned to the role.**

<ParamField path="id" type="string" required>
  The string ID of the role to delete.
</ParamField>

<Warning>
  The API will return an error if users are currently assigned to this role. Reassign or deactivate those users before deleting the role.
</Warning>

<CodeGroup>

```bash cURL
curl --request DELETE \
  --url 'https://your-api.example.com/api/usuarios/roles/TECNICO' \
  --header 'Authorization: Bearer <token>'
```

</CodeGroup>

### Sync modules for a role

`POST /api/usuarios/modulos/sync`

Replaces the full set of modules assigned to a role. Any modules not included in `moduloIds` will be unlinked from the role.

<ParamField body="rolId" type="string" required>
  The role ID to update module access for (e.g., `ADMIN`).
</ParamField>

<ParamField body="moduloIds" type="number[]" required>
  Complete list of module IDs to assign to the role. This is a full replacement, not an append.
</ParamField>

<Warning>
  This operation replaces the entire module list for the role. Any previously assigned modules not included in `moduloIds` will lose access.
</Warning>

<CodeGroup>

```bash cURL
curl --request POST \
  --url 'https://your-api.example.com/api/usuarios/modulos/sync' \
  --header 'Authorization: Bearer <token>' \
  --header 'Content-Type: application/json' \
  --data '{
    "idRol": "TECNICO",
    "moduloIds": [1, 3, 5]
  }'
```

</CodeGroup>

## Permission bitmask reference

The `permisos` field in a role is a `Long` integer where each bit value maps to a specific permission. To grant multiple permissions, sum their bit values.

| Bit value | Permission | Description |
|-----------|------------|-------------|
| 1 | `REGISTRAR_PENDIENTE` | Create new work orders |
| 2 | `EDITAR_PENDIENTE` | Edit existing work order fields |
| 4 | `VER_DETALLE_PENDIENTE` | View full detail of a single work order |
| 8 | `VER_TODOS_PENDIENTES` | List and search all work orders |
| 16 | `ASIGNAR_TECNICO` | Assign a technician to a work order |
| 32 | `ASIGNAR_PPOE` | Assign PPPoE credentials |
| 64 | `ASIGNAR_VLAN` | Assign VLAN configuration |
| 128 | `COMENZAR_TRABAJO` | Start field work (move to EN_PROGRESO) |
| 256 | `PARAR_TRABAJO` | Pause field work |
| 512 | `CONTINUAR_TRABAJO` | Resume paused field work |
| 1024 | `FINALIZAR_TRABAJO` | Mark field work as finished |
| 2048 | `VER_PENDIENTES_HISTORIAL` | View all work orders in history |
| 4096 | `REVISAR_GASTOS` | Review associated expenses |
| 8192 | `REVISAR_FINALIZADOS` | Review finalized work orders |

**Example calculations:**

- Full admin access: `1 + 2 + 4 + 8 + 16 + 32 + 64 + 128 + 256 + 512 + 1024 + 2048 + 4096 + 8192` = **16383**
- Technician (start/stop work + view detail): `128 + 256 + 512 + 1024 + 4` = **1924**
- Read-only viewer: `4 + 8 + 2048` = **2060**

<Tip>
  To check whether a user has a given permission in your frontend code, use a bitwise AND: `(permisos & 128) !== 0` returns `true` if `COMENZAR_TRABAJO` is granted.
</Tip>
